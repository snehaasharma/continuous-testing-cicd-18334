# Day 2: Test Automation Frameworks for CI/CD

**Session Duration:** 2 Hours  
**Focus:** 45% Theory | 55% Hands-on  
**Goal:** Design automation frameworks for CI execution, apply data-driven and CI-friendly patterns, and wire up build tools for terminal test execution.

---

## Part 1: Instructor Live Demo Script (Approx. 40-45 mins)

*Instructors: Follow this guide to demonstrate automation framework structure, CLI execution, and test reporting configuration.*

### Demo 1: Framework Project Structure Walkthrough

**Goal:** Demonstrate a CI-ready automation framework layout using Java/Maven.

1. **Context Setting:** 
   *(Instructor Script)*: "A well-structured framework separates UI interactions from test logic. We'll explore a Hybrid Framework pattern utilizing the Page Object Model, with tests driven by external data."
2. **Walkthrough:**
   - Open the starter project in your IDE.
   - Navigate through directories:
     - `src/main/java/pages/`: Page Object Classes.
     - `src/test/java/tests/`: Test classes (JUnit 5 / TestNG).
     - `src/test/java/utils/`: shared base classes.
     - `src/test/resources/`: `config.properties`, test data, suite configs.
   - Explain how configuration data (URLs, credentials) is loaded from properties, avoiding hardcoding explicitly for CI/CD.

### Demo 2: Executing Tests via Command Line (Maven/Gradle)

**Goal:** Execute tests via CLI, demonstrating the mechanism a CI agent will use.

1. **Context Setting:**
   *(Instructor Script)*: "In CI, there is no IDE graphical interface. Everything runs headlessly from the terminal using build tools like Maven or Gradle."
2. **Run the Full Suite:**
   - Execute: `mvn test` (or `./gradlew test`)
   - Show the console output, summarizing passes, failures, and skipped tests.
3. **Run Tagged Subsets:**
   - Execute: `mvn test -Dgroups=smoke` 
   - Highlight the importance of subset execution in pipelines (e.g., running smoke tests on PRs, full tests nightly).
4. **Exit Codes:**
   - Introduce a deliberate assertion error in a test.
   - Run `mvn test`, point out the `BUILD FAILURE` and explain the non-zero exit code.

### Demo 3: Generating Test Reports

**Goal:** Explain how automated tests communicate results securely and cleanly to developers.

1. **Locate Native Reports:**
   - Show the `target/surefire-reports/` directory where standard XML reports generated by the Surefire plugin reside.
   - These XML files are consumed automatically by Jenkins or GitHub Actions.

## Part 2: Participant Hands-on Lab (Approx. 65-75 mins)

**Welcome to the Day 2 Lab!** You will configure a test automation project, run it via the command line, and use tags to selectively execute a test suite, mimicking exactly what a CI server will do.

### Task 1: Set Up a Basic Automation Framework

1. Clone the sample course repository (if not already done).
2. Import the project into your IDE (IntelliJ or Eclipse) as a Maven or Gradle project.
3. Open your terminal at the project root and resolve dependencies:
   ```bash
   mvn dependency:resolve
   ```
4. Explore the `src/test/java` package. Familiarize yourself with how the basic tests are structured.

### Task 2: Configure Build Tool Integration

1. Open your `pom.xml` (or `build.gradle`).
2. Add the appropriate plugin configuration (e.g., Maven Surefire) to explicitly specify execution.
3. **Parallel Configuration:** Add settings to run tests in parallel, setting the thread count to 2.
4. **Tagging:** Go to one of your test classes and tag a test method using `@Tag("smoke")` (JUnit 5) or `@Test(groups="smoke")` (TestNG).

### Task 3: Execute Tests Locally via CLI & View Report

1. Run the full test suite from your terminal:
   ```bash
   mvn test
   ```
2. Now, run only your tagged smoke test:
   ```bash
   mvn test -Dgroups=smoke
   ```
   *(Note: for JUnit 5, you may need to configure `<groups>smoke</groups>` in pom.xml, or pass `-Dgroups=smoke` depending on your surefire setup).*
3. Add a deliberate error to your code (e.g., change an assertion).
4. Run tests again. Confirm that the terminal returns a non-zero exit code indicating test failure.
5. Locate the generated XML/HTML report in the `target/surefire-reports` or `build/reports` folder and document a screenshot.
